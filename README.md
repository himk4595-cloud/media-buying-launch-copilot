# Agentic Media Buying Launch Copilot

A launch workflow assistant for media buying teams.

The app turns a campaign brief into a structured launch packet with campaign naming, ad group strategy, audience angles, ad copy, landing page hooks, UTM links, deterministic QA checks, and export-ready JSON/CSV files.

## Live Demo

- Live app: **[https://media-buying-launch-copilot.vercel.app/packet]**
- GitHub repository: https://github.com/himk4595-cloud/media-buying-launch-copilot

## What does this tool do?

Media buyers often move from a rough campaign idea to launch setup through a mix of documents, spreadsheets, manual naming, copy drafting, tracking-link creation, and repeated QA.

This tool compresses that workflow into one guided process:

1. Enter a campaign brief.
2. Generate a launch-ready campaign packet with AI.
3. Review campaign structure, audience angles, headlines, primary text, landing page hooks, and UTM links.
4. Run deterministic pre-flight QA.
5. Export the packet as JSON or CSV for handoff or future platform integrations.

### Current workflow

```text
Campaign Brief
      ↓
Groq AI Generation
      ↓
Structured Launch Packet
      ↓
Pre-Flight QA Review
      ↓
JSON / CSV Export
```

### What the launch packet includes

- Campaign naming convention
- Ad group structure
- Audience angles
- Headlines
- Primary text options
- Landing page hook
- UTM-tagged landing page links
- Export payload
- Generation source indicator

### What the QA engine checks

- Missing required fields
- Campaign naming structure
- Daily budget range
- Landing page URL format
- Required UTM parameters
- Risky marketing language
- Missing compliance notes
- Headline length

## Why did I build this one?

I chose campaign launch preparation because it is a repeated, operationally important workflow for media buying teams.

A launch can be delayed or weakened by small mistakes:

- inconsistent naming
- incomplete briefs
- broken tracking parameters
- weak or repetitive creative angles
- risky claims
- missing compliance notes
- unclear handoff between strategy and execution

These problems are individually small, but they compound when a team launches many campaigns across multiple platforms.

The Launch Copilot is designed to improve two things that directly matter to a performance marketing business:

### 1. Launch speed

The app creates a structured first draft of the campaign setup in seconds instead of forcing a media buyer to assemble every element manually.

### 2. Launch reliability

The AI generation step is paired with deterministic QA. The tool does not simply generate copy; it checks the packet for operational issues before export.

That combination is intentional. Generative AI is useful for producing options, while rule-based validation is more dependable for checking required fields, tracking structure, and budget rules.

## Reliability and fallback behavior

The app attempts to generate the launch packet through Groq.

If the AI provider is unavailable, rate-limited, or returns an error, the app uses a deterministic fallback generator. This keeps the core workflow demonstrable and prevents the product from becoming unusable because of a third-party API failure.

```text
Try Groq
   ↓
Success → AI-generated launch packet
   ↓
Failure → Deterministic fallback packet
```

This fallback is not presented as AI. The Launch Packet page indicates whether the packet was generated by Groq AI or the fallback engine.

## What would I build next if this were my full-time job?

The current version stops at a reviewed, exportable launch packet. The next step would be to connect that packet to real media buying workflows.

### 1. Paused campaign creation

I would start with one ad platform and create campaigns in a paused or draft state.

The workflow would become:

```text
Generate Packet
      ↓
Run QA
      ↓
Human Approval
      ↓
Create Paused Campaign
      ↓
Final Review
      ↓
Activate
```

I would not allow AI-generated campaigns to spend money without a human approval gate.

### 2. Platform-specific connectors

I would add connectors for:

- Meta Marketing API
- Google Ads API
- TikTok Marketing API
- Taboola Backstage API

The generic launch packet would be translated into each platform's campaign, ad group/ad set, creative, ad, budget, and tracking structure.

### 3. Creative asset support

The next version would support:

- image uploads
- video uploads
- creative previews
- dimension and aspect-ratio checks
- file-size validation
- platform-specific creative requirements

### 4. Landing page inspection

The tool could crawl the destination page and verify:

- the URL resolves
- required disclosures are present
- the page matches the ad message
- tracking parameters survive redirects
- conversion pixels are present
- mobile rendering is usable

### 5. Performance feedback loop

After launch, the system could pull performance data and help the team:

- detect weak ads
- identify winning angles
- generate new variations
- recommend budget changes
- flag tracking anomalies
- summarize campaign performance
- create test plans based on live results

### 6. Team workflow and approvals

A production version would include:

- user accounts
- saved campaigns
- approval states
- comments
- version history
- audit logs
- role-based access
- campaign templates by vertical

## How the AI layer works

The browser sends the campaign brief to a server-side Next.js API route.

The route:

1. reads the campaign brief
2. builds a structured media buying prompt
3. requests JSON output from Groq
4. validates and parses the response
5. returns a launch packet to the browser

The API key remains server-side and is stored in an environment variable.

## Local setup

### Requirements

- Node.js
- npm
- A Groq API key

### Installation

```bash
git clone https://github.com/himk4595-cloud/media-buying-launch-copilot.git
cd media-buying-launch-copilot
npm install
```

Create a `.env.local` file in the project root:

```text
GROQ_API_KEY=your_groq_api_key_here
```

Start the development server:

```bash
npm run dev
```

Open:

```text
http://localhost:3000
```

### Production build

```bash
npm run build
```

## Demo walkthrough

1. Open the dashboard.
2. Click **Create launch packet**.
3. Enter a campaign brief or load the sample brief.
4. Click **Generate launch packet**.
5. Review the generated campaign structure, copy, landing page hook, and UTM links.

## Current limitations

- No direct ad platform integrations
- No real campaign creation or spend activation
- No database or user accounts
- Campaigns are stored locally in the browser
- QA rules are intentionally limited to the MVP
- AI output still requires human review
- No live campaign performance data
- No creative image/video upload workflow

## Safety principles

A production version should never allow AI-generated campaigns to activate spend automatically.

The intended safety model is:

- create drafts first
- validate tracking and required fields
- require human approval
- launch paused by default
- maintain audit logs
- restrict access by role

## Why the current scope is intentional

The goal of this version is to prove the workflow and business value before adding platform complexity.

The MVP demonstrates that a media buyer can move from an unstructured brief to a structured, QA-reviewed, exportable launch packet through one interface.

The next engineering milestone is not “more generated text.” It is connecting the reviewed packet to real campaign creation and performance data.
